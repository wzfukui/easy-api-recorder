# 为什么我要做一个“接口抓包助手”？

最近一段时间，我们在不少项目现场都会遇到同一类问题：客户已经有业务系统，也已经能在浏览器里正常跑起来，但一旦要把这些页面里的真实请求整理成可复用的 API 工具，整个过程就会立刻变得非常手工。

过去的常见流程通常是这样的：

1. 打开浏览器开发者工具。
2. 手动抓包。
3. 找到关键请求。
4. 复制请求头、响应头、请求体、响应体。
5. 再把这些信息转成脚本、测试代码，或者交给大模型继续生成。

这个流程不是不能做，而是非常消耗精力。它的问题不在于“技术难度高”，而在于“重复劳动太多”。尤其在现场支持场景里，工程师更需要的是效率，而不是一遍遍地在 F12 面板里复制、整理、排除干扰项。

所以我做了这个工具：`easy-api-recorder`。

## 这个工具解决什么问题？

它的核心目标很简单，就是把“浏览器里的真实网络请求”尽可能快地整理成“后续可以自动化利用的结构化资产”。

换句话说，它不是一个通用型抓包平台，也不是一个完整的测试平台，而是一个更贴近现场工程师工作流的小工具：

点开始录制 -> 走一遍业务流程 -> 筛选真正有价值的请求 -> 导出 JSON 或 HAR -> 交给模型或脚本继续开发

这样一来，很多原本要靠人工反复抄写、理解、组装的工作，就可以被压缩成一个更短的链路。

## 我为什么选择做成 Chrome 扩展？

因为现场工程师最熟悉的环境就是浏览器，而不是一个额外安装的大型桌面工具。

这个项目最终采用的是 Chrome Manifest V3 扩展方案，核心基于 `chrome.debugger`。这么做有几个明显好处：

- 可以直接附着到当前标签页，从真实业务页面里采集请求。
- 能够拿到更完整的网络事件生命周期，而不是只看到零散信息。
- 更适合和现场工作流结合，不需要额外教用户一套全新的工具。

从使用角度看，它保留了浏览器抓包的“原地操作”优势；从开发角度看，它又比纯手工用 F12 更容易做自动化整理和导出。

## 这个工具现在大概能做什么？

目前这版已经能覆盖一个相当实用的 MVP 场景：

- 选择当前标签页开始录制。
- 通过 `chrome.debugger` 监听网络请求。
- 支持按资源类型抓取，例如 `Document`、`XHR`、`Fetch`，也可以按需扩展到图片、脚本、样式等静态资源。
- 列表展示抓到的请求，并支持搜索、类型筛选、批量勾选。
- 点击条目后查看详情，包括请求头、响应头、请求体、响应体。
- 支持把选中的请求加入导出队列，减少干扰项。
- 支持导出 JSON 或 HAR，便于后续交给大模型、测试脚本或接口工具链继续处理。

对于现场支持来说，这已经足够把“抓包 -> 理解 -> 转代码”的前半段工作显著提速。

## 技术上我是怎么做的？

这套工具的整体思路其实比较克制，没有追求特别重的架构，而是优先保证可用性和清晰度。

大体上可以分成四块：

### 1. 录制层

录制层由后台脚本负责，核心职责是附着标签页、启动网络监听、收集请求生命周期，并把同一个请求的不同阶段拼装成统一结构。

### 2. 规则与过滤层

为了避免抓出来的内容全是噪音，我在工具里加了资源类型控制、URL 包含规则、URL 排除规则，以及是否抓取二进制响应体等设置。这样工程师在现场就能尽量只保留真正有价值的内容。

### 3. 浏览与筛选层

结果页负责把抓到的数据用更适合人工浏览的方式展示出来。这里不是简单把所有包平铺，而是增加了搜索、类型筛选、详情抽屉、导出队列等能力，让“找重点”这件事变得更快。

### 4. 导出层

内部先保留一份更容易处理的结构化 JSON，再根据需要转换成 HAR。这样既能兼容传统抓包文件格式，也方便后续交给大模型直接理解。

## 实际效果怎么样？

我觉得这个工具最直接的价值，不是“抓得更多”，而是“让有效信息更快浮出来”。

以前很多时候，工程师在浏览器里抓到一堆请求之后，真正耗时的是：

- 哪几个请求才是关键的？
- 哪些是静态资源、哪些是干扰项？
- 哪些请求值得导出？
- 导出去之后怎么继续给模型或者脚本使用？

现在这几个问题，至少在工具层已经有了一个更顺手的答案。用户可以先抓，再筛，再加入导出队列，最后统一导出。整个流程会比纯人工复制粘贴稳定得多。

## 接下来我还想怎么迭代？

这个工具现在已经能用，但我觉得还有几个方向值得继续做：

### 1. 更强的规则能力

比如支持更细粒度的匹配逻辑、规则模板、常见场景预设，让现场工程师能更快进入可用状态。

### 2. 更好的导出语义

目前重点是导出原始请求信息，后续可以继续补“适合大模型直接消费”的语义化结构，例如把鉴权、分页、列表请求、详情请求等模式整理出来。

### 3. 更稳定的二进制与大响应体处理

静态资源、二进制内容、超大 body 的处理还可以继续优化，尤其是体积控制和可读性之间的平衡。

### 4. 更完整的分享与协作能力

如果后面需要，也可以考虑把一次抓包结果做成更适合团队共享的记录格式，而不只是本地导出文件。

## 最后

我做这个工具，并不是想替代浏览器开发者工具，而是想把现场工程师最频繁、最重复、最容易出错的那一小段工作流单独拿出来，做得更顺手一点。

如果一个工具能把“点 F12、复制请求、手抄脚本、反复检查”变成“点开始、走业务、筛重点、导出结果”，那它就已经有存在的价值了。

而 `easy-api-recorder`，就是沿着这个方向做出来的一个小而实用的工具。
